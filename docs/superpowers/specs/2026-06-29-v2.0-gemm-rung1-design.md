# v2.0 GEMM — Rung 1 (Foundation + De-risk) — Design

**Status:** Approved (brainstorming, 2026-06-29)
**Increment:** v2.0 GEMM/matmul — the first "hard" kernel class (v1 explicitly
excluded it). Branch `v2.0/gemm` off `main` (v1.1.0).
**Hardware:** A100 (sm_80). Baseline for GEMM = `torch.matmul` (cuBLAS) via the
existing `torch.compile(reference)` machinery — no perf-stage change needed.

## Overall GEMM goal (the direction these rungs build toward)
**Fused GEMM epilogues** — `activation(a @ b + bias)` in one kernel — are the
winnable, mission-aligned target: they beat torch's matmul-then-separate-
elementwise baseline by saving a result round-trip, even when the inner GEMM is
merely *decent* (not cuBLAS-class). Beating cuBLAS on a bare GEMM is not winnable
and is explicitly NOT the goal. Built in rungs, each proven before the next:
1. **Rung 1 (this spec): foundation + de-risk** — can the engine generate a
   *correct* tiled GEMM at all? (Never tested — all of v1/v1.1 was elementwise/
   reductions/scans.)
2. Rung 2: fused epilogue (the value) + systematized GEMM codegen guidance +
   rectangular shape support if needed.
3. Rung 3: tensor-core (WMMA/MMA) inner GEMM + tile-shape autotuning.

## 1. Rung 1 goal
Find out — cheaply and honestly — whether the engine can synthesize correct,
reasonable tiled GEMM kernels, and what fraction of cuBLAS they reach, *before*
investing in fused epilogues or engine-wide codegen changes. The experiment is a
small set of GEMM eval fixtures with strong prompts, run on Colab.

**Done =** a `gemm` eval suite of hand-written fixtures exists and is structurally
validated locally; a credited Colab A100 run records, per fixture: compiles?
correct vs `torch.matmul`? speedup vs cuBLAS. An evidence doc captures the result
and a go/no-go + shape decision for Rung 2.

## 2. Why no engine code changes in Rung 1
The de-risk must be cheap and not commit us prematurely. Two facts make that
possible:
- **Square GEMM needs no correctness-machinery change.** The input generator
  binds symbolic dims positionally per-arg; for `a:(N,N)`, `b:(N,N)` with a square
  fallback shape, both inputs become `N×N` and `a @ b` is valid. (Rectangular
  M≠K≠N would need cross-arg symbol binding — deferred to Rung 2.)
- **The GEMM-specific guidance lives in the fixture prompts**, not the codegen
  prompt. Each fixture's `prompt.txt` tells the LLM to write a shared-memory-tiled
  GEMM (and may mention the `sm_80` tensor-core MMA shapes already in target
  caps). If the spike shows this works, Rung 2 systematizes the guidance into the
  codegen stage; if it struggles, we learn exactly what's missing. No premature
  engine change.

## 3. Components
- **`evals/gemm/` suite** (new), reusing the existing fixture format
  (`prompt.txt`, `reference.py`, `shapes.yaml`, `notes.md`):
  - `matmul_fp16` — `reference(a, b): a @ b`, square shapes; prompt asks for a
    correct shared-memory-tiled fp16 GEMM. The core correctness de-risk.
  - `matmul_fp32` — same in fp32 (CUDA-core tiling, no tensor cores).
  - `matmul_bias_gelu_fp16` — `gelu(a @ b + bias)`, the fused-epilogue probe (the
    eventual value target); square; bias broadcast over rows.
  - (2–4 fixtures total — small on purpose.)
  - All **square** (M=K=N) so the existing input generator produces compatible
    matmul shapes.
- **`cuda-engine eval --suite gemm`** — add `"gemm"` to the CLI suite-name map
  (one line); the runner already handles any suite directory.
- **Evidence doc** `docs/milestones/v2.0-gemm-rung1-evidence.md` — filled from the
  Colab run.

## 4. Testing
- Unit (local, no hardware): a suite test mirroring `test_internal_eval_suite` —
  fixtures discoverable, required files present, ≥1 shape each, references import
  and produce finite outputs under the input generator, names don't collide with
  internal/kernelbench. Verify the bias fixture's shapes broadcast correctly.
- Hardware (credit-gated): the Colab spike itself is the integration test;
  results recorded in the evidence doc.

## 5. Decision gate (end of Rung 1)
From the spike: (a) if correctness is reliable and the fused probe shows promise →
proceed to Rung 2 (fused epilogues + systematized guidance, rectangular shapes).
(b) if correctness is shaky → Rung 2 becomes "make tiled-GEMM correctness
reliable" (prompt/codegen work) before fusion. Either way the spike, not a guess,
drives the next spec.

## 6. Out of scope (Rung 1)
- Engine codegen-prompt changes (Rung 2, informed by the spike).
- Rectangular (M≠K≠N) shapes / cross-arg symbol binding (Rung 2 if justified).
- Tensor-core WMMA codegen as a requirement, tile autotuning (Rung 3).
- Attention (v3), batched/strided GEMM, multi-GPU.
