# v1.1 "Sharpen" — Design

**Status:** Approved (brainstorming, 2026-06-28)
**Increment:** #1 of the post-v1.0 sequence (#1 sharpen → #4 hardening → #2 GEMM).
**Builds on:** v1.0.0 (shipped to PyPI 2026-06-28). See
`docs/milestones/M4-evidence.md` and the post-v1 roadmap in
`2026-04-26-cuda-synthesis-engine-design.md` §"Post-v1 roadmap".
**Hardware:** A100 (sm_80) only — no new GPU dependency.

## 1. Goal & success criteria

Make the shipped engine genuinely sharper without expanding hardware or the
kernel scope. Three parts, shipped as one release:

1. **Multi-provider LLM layer** — run the engine on any provider, with Claude
   still the capability-rich default, and benchmark models against each other.
2. **Fix the two slow kernels** — `gelu_fp16` (0.67×) and `rms_norm_fp16`
   (0.75×) to ≥1.0× vs torch.compile.
3. **Broaden eval coverage** — add in-scope kernels so the provider benchmark
   is meaningful.

**Done =**
- Providers are swappable behind one interface; Claude's prompt caching + tool
  use remain first-class (never leveled down).
- `cuda-engine eval` can run the same suite across ≥3 providers and emit a
  "which model writes the best CUDA" comparison (functional % + median speedup
  + tokens/cost per model).
- `gelu_fp16` and `rms_norm_fp16` both reach ≥1.0× (or are documented as a real
  roofline limit — no faking).
- Full unit suite + ruff + mypy green; one credited Colab A100 run validates the
  provider sweep and the two kernel fixes.

## 2. Multi-provider LLM architecture (core)

Principle (from the user): **best quality, no bandaid, flexible with any
provider.** This rules out routing Claude through a lowest-common-denominator
API (a bandaid on quality) *and* rules out hand-writing an adapter per provider
forever (not flexible). The design is **capability-aware and native-first**,
built on the existing `LLMClient` seam (design doc §"Layer rule").

### 2.1 Provider interface
- `LLMProvider` exposes `complete(messages, tools, cache_hints, model, …)` plus
  a **`capabilities`** descriptor: `prompt_caching`, `tool_use`, `max_context`,
  etc.
- Stages request features (caching, tools). A provider either honors them
  natively or the stage **degrades gracefully and the run report records the
  downgrade** — nothing silently breaks, and the Claude path is never dumbed
  down.

### 2.2 Adapters
- `AnthropicProvider` — exists; prompt caching + tool use; **stays the default**
  (Sonnet default → Opus escalation, unchanged).
- `OpenAIProvider` — native function calling + automatic prompt caching.
- `GeminiProvider` — native tool calling + context caching.
- `OpenAICompatibleProvider` — **one** generic adapter pointed at any `base_url`
  speaking the OpenAI API. This is the "any provider" piece done properly:
  OpenRouter, Together, Groq, DeepSeek, vLLM/local models all reachable through
  a real, widely-adopted protocol — not a fake uniform shim.

### 2.3 Per-stage model routing
- Config maps each of the 5 stages → `(provider, model)`. Default unchanged.
- Lets cheap stages (e.g. Interview) route to cheap models while hard stages
  keep a strong model.

### 2.4 Provider sweep in the eval harness
- `cuda-engine eval --provider … --model …` plus a sweep matrix runs the same
  suite across providers.
- Summary compares **functional % + median speedup + tokens/cost** per
  provider/model. This is the benchmarking feature, built on the existing
  resumable eval harness.
- **Cost-aware:** the sweep is opt-in with a cost preflight; defaults to a small
  subset unless the full matrix is requested.

### 2.5 Honest scope flag
The LLM layer is large — arguably its own version. If it balloons during
implementation, split the benchmarking sweep (§2.4) into a v1.2 and ship
§2.1–2.3 first.

## 3. Slow-kernel fix (investigation-first)

The data already points at codegen, not physics: `gelu_fp16` 0.67× vs its
sibling `bias_gelu_fp16` 1.03× (which does *more* work), and `rms_norm_fp16`
0.75× vs `rmsnorm_silu_fused_fp16` 1.04×. That asymmetry says the generated
kernel is the problem (likely exact-`erf` GELU vs fast tanh approx, missing
`half2`/`float4` vectorization, or extra memory traffic).

**Approach (systematic-debugging, no blind prompt-tweaking):**
1. Pull the generated `.cu` + `stage4_performance/.../nsight.json` (+ `report.json`
   stage traces) for both kernels from the run artifacts on Drive.
2. Read Nsight counters (memory throughput vs HBM peak, occupancy, instruction
   mix) to pin the root cause.
3. Apply a fix at the **codegen/prompt or perf-repair** layer that **generalizes**
   to the op class — not a hand-edit of two `.cu` files.
4. Re-run both on a credited Colab A100; gate = both ≥1.0×.
5. **Fallback:** if it's genuinely a roofline limit, document it and drop the
   target rather than fake it.

## 4. Eval expansion + testing/verification

- **More kernels:** add ~10–12 in-scope kernels (elementwise / fused /
  reductions / scans — **no** GEMM/attention; those are increment #2). Reuse the
  existing fixture format + the no-overlap test.
- **Testing:** unit tests per provider adapter (mocked API responses),
  capability-degradation tests, eval-sweep tests (mocked, no real calls).
  Integration tests stay `@integration`-gated.
- **Verification before done:** full unit suite + ruff + mypy green locally,
  then one credited Colab run proving (a) the provider sweep emits a comparison
  report and (b) both kernels hit ≥1.0×.

## 5. Out of scope (explicit)
- GEMM / matmul / attention (increment #2).
- New hardware: sm_90 / sm_100 (needs rented hours; not this increment).
- Run resumability / persistent run DB / autotuning sweeps (increment #4).
