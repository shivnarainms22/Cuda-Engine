# v1.1a â€” Multi-Provider LLM Layer Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let the engine run on any LLM provider behind one interface â€” Claude stays the capability-rich default â€” with per-stage model routing and a provider-benchmark sweep in the eval harness.

**Architecture:** Introduce an `LLMRouter(LLMClient)` that holds a registry of provider adapters keyed by name and dispatches `complete()` on a namespaced model id (`"<provider>:<model>"`). Each non-Anthropic adapter translates the existing Anthropic-shaped `system`/`messages`/`tools` payload into its native API and maps the response back to `LLMResponse`. A `capabilities` descriptor per provider lets stages degrade gracefully (e.g. no prompt caching) while the run records the downgrade. The `Orchestrator` keeps injecting a single `LLMClient` (now the router); per-stage `(provider, model)` comes from `SynthesisConfig`.

**Tech Stack:** Python 3.11, pydantic v2, `anthropic` SDK (existing), `openai` SDK (new), `google-genai` SDK (new), pytest. Mocked-SDK unit tests; real calls only under `@integration`.

**Spec:** `docs/superpowers/specs/2026-06-28-v1.1-sharpen-design.md` Â§2, Â§4.
