# v1.1b — Slow-Kernel Fix (bound-aware perf repair) Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Lift `gelu_fp16` (0.67×) and `rms_norm_fp16` (0.75×) to ≥1.0× vs torch.compile by making the Stage-4 performance-repair loop *bound-aware* — a generalizable fix, not a hand-edit of two `.cu` files.

**Architecture:** The repair loop currently sees only `occupancy/regs/spill` and defaults to "add more ILP," which lowers occupancy and hurts latency-bound kernels. We (1) extract the bottleneck-revealing Nsight signals already present in the ncu CSV (`memory throughput %`, `DRAM %`, `compute %`, `waves/SM`, achieved bandwidth, SOL bottleneck rule), (2) classify the bound type (latency / bandwidth / compute), and (3) feed that diagnosis + targeted guidance into the repair prompt. Then re-run the two kernels on Colab to confirm ≥1.0×.

**Tech Stack:** Python 3.11, pydantic v2, pytest. Real ncu-CSV fixtures (captured from the v1.0 run) drive the parser tests — no GPU needed for unit tests.

**Spec:** `docs/superpowers/specs/2026-06-28-v1.1-sharpen-design.md` §3.

**Diagnosis evidence (committed fixtures):** `evals/fixtures/nsight/gelu_fp16/*.json`, `.../rms_norm_fp16/*.json` (the per-attempt nsight.json captured 2026-06-28).

---

## Findings that drive this plan
- `gelu_fp16`: latency-bound (DRAM 21–36%, occupancy 36–66%); repair *increased* regs (30→64) and *lowered* throughput. Sibling `bias_gelu_fp16` (more work) hits 1.03×, so ≥1.0× is reachable via a high-occupancy, low-register float4 grid-stride mapping with correct grid sizing.
- `rms_norm_fp16`: best attempt hit 86% DRAM but does **2 input reads**; a separate attempt register-cached the input (1 read) but launched 0.79 waves/SM. Combining caching + full-GPU launch cuts memory traffic ~⅓ → ≥1.0× expected.
